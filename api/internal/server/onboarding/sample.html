Summary

On Sept 14, the payments API returned 5xx errors for roughly 12 minutes during the impact window, affecting 3.2% of merchants. The root cause was a stale connection being returned from the upstream pool after the idle timeout elapsed mid-request[1].

Root cause

The connection pool's idle-timeout check ran before the context deadline check, so connections that had just crossed the idle threshold were returned for active requests and immediately rejected by the upstream. 

if conn.IdleTimeout < ctx.Deadline() {
  // pool returned a stale conn → 5xx
  return ErrUpstream
}
The fix (one-line reorder) 
if ctx.Deadline() // check deadline first
   && conn.IdleTimeout > remaining(ctx) {
  return pool.Drop(conn)
}

Action items

  • Reorder pool checks: deadline first, then idle timeout.
  • Add a synthetic probe for stale-conn behavior on every release.
  • Raise alert sensitivity for payment-success-rate dips below 99.5%.
